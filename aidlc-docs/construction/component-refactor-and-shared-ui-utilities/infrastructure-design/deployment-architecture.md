# Deployment Architecture - Component Refactor And Shared UI Utilities

## Deployment Architecture Summary

Unit 2 does not change deployment architecture. The refactored frontend modules are compiled by the existing Vite build and served as static assets through the existing GitHub Pages deployment model.

## Environments

| Environment | Role | Unit 2 Impact |
|---|---|---|
| Local development | Developer edits and previews React/Vite app | Refactored components and utilities run through the Vite dev server. |
| Local verification | Developer runs static checks | `npm run build` and `npm run lint` validate Unit 2 implementation. |
| GitHub Actions | Existing CI/deployment runner | No workflow changes in Unit 2; later Unit 3 owns deployment config changes. |
| GitHub Pages | Existing static host | Serves the built output after deployment. |

## Deployment Flow

1. Developer edits React components, shared UI modules, and utilities in the workspace root.
2. Developer runs `npm run build`.
3. Developer runs `npm run lint`.
4. Existing GitHub Actions workflow can continue to build and publish `dist/`.
5. GitHub Pages serves the static HTML, CSS, JavaScript, images, and PDFs.

## Unit 2 Build Artifacts

Unit 2 implementation contributes to the same static artifact set as the rest of the app:

- JavaScript bundle generated by Vite.
- CSS generated from existing CSS and Chakra usage.
- Vite-resolved image and PDF assets.
- Static `index.html`.

No additional deployment artifact type is introduced.

## Infrastructure Ownership Boundaries

| Concern | Owner |
|---|---|
| Component/data wiring | Unit 2 |
| Shared frontend utilities | Unit 2 |
| Shared UI helper lint fixes | Unit 2 |
| GitHub Pages base path | Unit 3 |
| GitHub Actions deployment workflow changes | Unit 3 |
| Student setup/deployment docs | Unit 4 |
| Formal tests and test command | Unit 5 |

## Deployment Constraints

- No server runtime is available on GitHub Pages.
- No environment secrets should be required by Unit 2.
- No API endpoint should be required for contact, navigation, media, or content rendering.
- Build and lint checks should be runnable locally and in CI without network-dependent validation.

## Rollback Considerations

Unit 2 changes are normal source-code changes in the static app. Rollback uses Git history and does not require infrastructure rollback, database migration rollback, queue draining, cache invalidation, or service redeployment.

## Future Deployment Work

Unit 3 will handle deployment configuration concerns such as Vite base-path behavior and GitHub Pages workflow improvements. Unit 2 intentionally avoids those changes to keep deployment architecture stable while UI internals are refactored.
